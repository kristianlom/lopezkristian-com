Cyclomatic complexity is the oldest and most reliable software metric. First proposed by Thomas McCabe in 1976, it counts the number of linearly independent paths through a piece of code. In 2025, it remains the single best leading indicator of code that will be painful to test, maintain, and understand.

## The Formula

For a function with:
- `d` decision points (if, else if, while, for, case, catch, &&, ||, ternary)
- The cyclomatic complexity CC = d + 1

```java
// CC = 1 (no decisions)
public String formatName(String first, String last) {
    return first + " " + last;
}

// CC = 4 (three decisions)
public String classify(int score) {
    if (score >= 90) return "A";        // +1
    else if (score >= 80) return "B";   // +1
    else if (score >= 70) return "C";   // +1
    return "F";
}
```

## Thresholds That Matter

| CC Range | Risk Level | Recommendation |
|----------|-----------|----------------|
| 1–5 | Low | Ideal |
| 6–10 | Moderate | Acceptable for complex algorithms |
| 11–20 | High | Mandatory refactoring target |
| 21–50 | Very High | Significant risk, prioritize |
| 50+ | Untestable | Rewrite |

## Measuring in Java

With Maven and the Checkstyle plugin:

```xml
<module name="CyclomaticComplexity">
  <property name="max" value="10"/>
  <property name="severity" value="error"/>
</module>
```

Or with PMD:
```xml
<rule ref="category/java/design.xml/CyclomaticComplexity">
  <properties>
    <property name="classReportLevel" value="80"/>
    <property name="methodReportLevel" value="10"/>
  </properties>
</rule>
```

## The Architectural Perspective

Cyclomatic complexity is a proxy for coupling. Functions with CC > 10 usually exhibit:

1. **Multiple responsibilities** — violates SRP
2. **Deep nesting** — indicates control flow that should be polymorphism
3. **Implicit state machines** — candidate for the State pattern
4. **Combinatorial testing burden** — CC = 15 means 15 test cases minimum

The refactoring moves:

```java
// Before: CC = 8
public BigDecimal calculateDiscount(Order order) {
    if (order.getCustomerType() == PREMIUM) {
        if (order.getTotal().compareTo(LARGE_ORDER) > 0) {
            return order.getTotal().multiply(new BigDecimal("0.20"));
        } else {
            return order.getTotal().multiply(new BigDecimal("0.10"));
        }
    } else if (order.getCustomerType() == VIP) {
        // ... more branches
    }
    return BigDecimal.ZERO;
}

// After: CC = 1 per strategy
public interface DiscountStrategy {
    BigDecimal apply(Order order);
}

public class PremiumDiscountStrategy implements DiscountStrategy {
    @Override
    public BigDecimal apply(Order order) {
        var rate = order.getTotal().compareTo(LARGE_ORDER) > 0 
            ? new BigDecimal("0.20") 
            : new BigDecimal("0.10");
        return order.getTotal().multiply(rate);
    }
}
```

## When to Ignore It

Cyclomatic complexity is a tool, not a law:
- **Parsers and state machines**: inherently complex, often appropriately so
- **Generated code**: don't waste time on it
- **Switch on discriminated unions**: acceptable pattern, low cognitive load despite high CC

The goal is to use complexity as a detection mechanism, not to game the metric.
